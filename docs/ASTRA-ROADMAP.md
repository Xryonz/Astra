## T4 — Desktop nativo + KMP  [FUTURO — depois da paridade mobile]
Meta: reaproveitar o Kotlin do /mobile-native no desktop sem reescrever.
- **KMP (Kotlin Multiplatform)** — mover `data/` + `domain/` (repos, use cases, models, DTOs) pra um modulo `shared` que compila pra Android + JVM Desktop. Comecar extraindo 1 feature pequena (auth) pro shared, validar nos 2 targets, migrar o resto. Tensao: Hilt e Retrofit sao Android-only (ver T6) — trocar antes.
- **Compose Multiplatform** — as telas Compose viram janela desktop quase sem mudar. Usar `org.jetbrains.compose`; APIs Android (Toast, Context, MediaProjection) viram `expect/actual`. StarField/CosmicBackground/Editorial migram direto (Canvas e multiplataforma).
- **Ktor client** — substitui o Retrofit (Android-only) na rede do shared. `HttpClient` + `ContentNegotiation(kotlinx.serialization)` + engine OkHttp (Android) / CIO (desktop). Pre-requisito real do T4; mantem os mesmos DTOs.
- **LiveKit Kotlin SDK** — no Android e `io.livekit:livekit-android` (ja usado, M6). ATENCAO: esse SDK e Android-only (WebRTC empacotado pra Android). Voz/video no desktop NAO sai de graca do Compose MP — sem SDK JVM oficial; opcoes: WebView com o JS SDK, ou processo nativo. Decidir antes de prometer call no desktop.
- **Conveyor (Hydraulic)** — gera instaladores assinados (.exe/.msi, .dmg, .deb) + auto-update do Compose Desktop. `conveyor.conf` aponta pra distribuicao do Compose Desktop; CI cospe os arquivos de download. Custo zero pra open-source.
- **Config desktop: aceleracao de hardware** — toggle nas Configuracoes do desktop pra render via GPU. Compose Desktop ja usa Skia; expor o backend de render (ex: `skiko.renderApi`) numa setting, com fallback software se a GPU bugar. [anotado]

## T5 — Features de produto  [BACKLOG]
- **Aba de plugins** — extensibilidade tipo marketplace interno. Definir API estavel (eventos onMessage, slash-commands, paineis) e rodar plugin em sandbox (JS via QuickJS/J2V8, ou modulo Kotlin assinado). Comecar read-only (plugin so le eventos) antes de dar escrita. Risco: seguranca — plugin de terceiro precisa de escopo de permissao, nao le tudo.
- **IA "resumir canal"** — botao que le e resume o que rolou enquanto voce esteve offline. Ja existe IA no backend (`apps/api/src/lib/bot.ts` usa Claude). Endpoint novo `POST /api/channels/:id/summarize?since=<lastReadAt>`: pega as msgs nao-lidas, manda pro Claude com prompt de resumo, devolve texto; mobile poe um botao no header do canal -> resumo num sheet. TENSAO: exige o servidor LER as msgs — incompativel com E2EE total (ver criptografia).
- **Modo LAN** — conversar/voz na rede local sem internet. Descoberta via mDNS/NSD (Network Service Discovery), um peer vira host efemero, msgs P2P. Escopo grande — comecar por "achar peers na LAN + chat texto"; voz LAN depois. Bom pra LAN party.
- **Eco-mode** — toggle que poupa bateria/CPU: pausa animacoes (StarField/Reveal), baixa polling/heartbeat, congela GIFs no 1o frame, reduz qualidade de video na call. Irmao do reduce-motion mas pra energia; liga sozinho em bateria baixa (`BatteryManager` <20%).
- **Swipe (Jetpack Compose)** — gesto pra economizar clique. `SwipeToDismissBox` / `anchoredDraggable` nas linhas: swipe na DM/canal -> acao rapida (responder/marcar lido/arquivar); swipe na msg -> responder. Editorial: revelar a acao com accent atras. Cuidado: nao brigar com scroll horizontal vizinho.
- **Melhorar a criptografia** — hoje: TLS em transito + bcrypt na senha. Proximo nivel = E2EE (msgs cifradas ponta-a-ponta) via libsignal (protocolo Signal) ou NaCl/age; chave no dispositivo (Keystore Android / equivalente desktop), servidor so guarda blob cifrado. TENSAO PESADA: quebra busca no servidor, resumo por IA e preview de link (tudo que LE a msg). Decidir: E2EE so em DM (canais legiveis pra IA/busca) ou abrir mao do resumo server-side.
